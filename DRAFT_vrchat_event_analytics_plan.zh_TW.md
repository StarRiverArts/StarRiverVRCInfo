# VRChat 活動分析系統草案（暫不實作）

> 狀態：草案  
> 備註：**不立刻實行，仍需進一步討論**。本文僅整理方向、用途與技術風險，不代表目前已有可行實作。

## 目的

規劃一套面向 VRChat 活動的分析系統，用來觀察活動中的玩家進出、停留、流量變化與活動效果。  
方向上可參考 VRCX 這類「讀取執行中 VRChat 客戶端資訊並做本地分析」的模式，但資料取得方式目前仍是最大難點。

## 預期想看到的分析

- 活動期間同時在線人數變化
- 玩家進場與退場時間分布
- 尖峰時段
- 平均停留時間
- 活動前、中、後的人流變化
- 不同場次的流量比較
- 活動開始後多久到達高峰
- 高峰維持時間
- 若能做到，還可延伸成：
  - 場內熱區推估
  - 主持節點前後人數波動
  - 單一玩家回訪率

## 可能的資料來源

### 1. 本機 VRChat 執行中資料

這是最接近需求的方向。  
若能從本機執行中的 VRChat 取得 instance / player join-leave 相關資訊，就能建立活動分析。

可能手段：

- 讀取本機 log
- 監聽 VRChat 產生的事件資訊
- 由外部工具讀取執行狀態
- 參考 VRCX 類似的本地資料聚合方式

### 2. 手動事件紀錄

若自動讀取難度太高，至少可以先接受：

- 活動開始時間
- 活動結束時間
- 主持人手動標記節點
- 場次名稱
- world / instance 識別資訊

這樣至少能建立半自動分析。

### 3. 活動方自建記錄

例如：

- 場控手動記錄
- 外部 bot 或觀測工具輸出
- 其他本地輔助程式

## 目前最大的困難

### 1. 資料取得合法性與可行性

最核心問題不是畫圖，而是**能不能穩定取得資料**。

需要先釐清：

- VRChat 客戶端本地有哪些資料可以讀
- 哪些資料是 log 層可得
- 哪些資料其實沒有官方穩定來源
- 是否存在版本更新後就失效的風險

### 2. 玩家進出事件不一定完整

即使能讀到部分本地資訊，也可能遇到：

- 只看到部分玩家
- 事件順序不穩定
- 切世界、斷線、重連造成誤判
- 同一玩家名稱變動或顯示不一致

### 3. 活動分析需要時間軸品質

這種系統的關鍵是時間序列。如果資料只有零碎快照，分析價值會大幅下降。

## 建議產品定位

這套系統不要一開始就定位成「精準統計每個玩家的一切行為」，那會過度擴張。  
比較合理的第一版定位是：

**本地活動觀測與回顧工具**

也就是：

- 觀察活動整體流量
- 回顧場次節奏
- 估算活動成效
- 對主辦有參考價值

## 建議資料模型草案

- `events`
  - 活動名稱、日期、主辦、世界、instance、備註
- `sessions`
  - 單次觀測 session
- `presence_events`
  - 玩家進場、退場、切換狀態
- `population_samples`
  - 某時刻的在線人數快照
- `timeline_marks`
  - 主持流程標記，例如開場、比賽、抽獎、合照

## 可做的畫面

- 活動總覽頁
- 單場活動時間軸頁
- 在線人數曲線
- 玩家停留分析
- 場次比較頁
- 事件標記與備註頁

## 技術路線草案

### 路線 A：先做手動 / 半自動版本

- 手動建立活動
- 手動標記事件節點
- 匯入本地時間序列資料
- 先完成視覺化與資料結構

優點：

- 可先驗證分析畫面是否真的有價值
- 不會卡死在資料抓取問題上

### 路線 B：本地觀測工具

- 在使用者本機執行一個 collector
- collector 讀取 VRChat 運行中的可得資料
- 再把資料匯入分析介面

優點：

- 更接近理想型

缺點：

- 技術風險高
- 需要處理版本變動與觀測可靠性

## 與 VRCX 類型工具的關係

這份草案的思路可參考 VRCX 類型工具的方向：

- 由本地端觀測資料
- 做事件與時間序列整理
- 用圖表協助人理解活動狀態

但目前不應直接假設：

- 一定能拿到完全相同的資料
- 一定能低成本複製其做法
- 一定適合直接部署成公開網站

更合理的模型可能是：

- 本地 collector
- 本地或私有 API
- Web 端只負責展示與分析

## 隱私與風險

這類系統比一般 world 資料分析更敏感，因為會碰到玩家 presence。

必須先討論：

- 是否只分析聚合數據，不保留個人明細
- 是否只限活動主辦自用
- 是否需要匿名化處理
- 哪些資料可以公開展示，哪些只能本地保存

## 目前不建議立刻做的事

- 直接假設可以穩定監控所有玩家進出
- 一開始就做公開站點
- 一開始就做很重的即時 dashboard
- 沒確認資料來源前就先做大量前端

## 建議討論順序

1. 先釐清想分析的是活動整體，還是玩家個體
2. 再釐清可取得的資料來源
3. 再決定先做手動版還是本地 collector 版
4. 最後才決定網站化程度

## 待討論問題

- 最小可接受的資料來源是什麼？
- 是否能接受先以 log / 手動標記為主？
- 目標是活動回顧，還是接近即時監看？
- 是否需要保留玩家層級資料？
- 是否只限本地使用，而非直接公開上站？
